# 08 — Architecture and deployment

## Architecture objective

The current build prioritizes an inspectable jury demonstration and a low-risk
industrial starting point:

- public static demonstration plus an offline-capable local archive;
- zero application dependency and no external AI or telemetry API;
- deterministic constraint planning and KPI calculations;
- a separate embedded learned assist that never changes a schedule;
- explicit separation between data, planning, recommendation and presentation;
- human approval and no machine or production-system control.

## Current end-to-end architecture

```text
engine/factory-data.js
Synthetic lines, orders, schedule, incident and coefficients
                         |
                         v
engine/twin-engine.js
Validation -> incident analysis -> 3 constrained plans -> KPI
                         |
             +-----------+------------+
             |                        |
             v                        v
engine/recommender.js       engine/exact-benchmark.js
Suggests one existing      Separate exhaustive bounded
route from local model     verification path
             |                        |
             +-----------+------------+
                         |
                         v
app/app.js + app/factory-view.js
Decision view <-> isometric workshop twin
                         |
                         v
Human preview -> route choice -> approval event -> reset
```

The deterministic planner remains the source of all three schedules and KPI.
The recommender only classifies the incident and those completed plans. The
exact verifier does not reuse the live planner's construction logic, although
it deliberately shares canonical scenario validation and KPI definitions.

## Offline training path

```text
Twin-generated synthetic incident grid
stopped line x start time x duration
                         |
                         v
Deterministic planner generates 3 routes per valid incident
                         |
                         v
Documented arbitration policy assigns one label
                         |
                         v
Seeded softmax training -> quality gates -> generated local model
```

The current model uses **687 simulated incidents generated by the twin itself**,
16 explainable features and a seeded 515/172 train/test split. It records 95.7%
training accuracy and 93.6% held-out synthetic-grid test accuracy. No real
factory or customer data is used.

## Data and decision flow

1. `createFactoryScenario()` returns a new controlled scenario.
2. `validateScenario()` checks IDs, ranges, compatibility, durations, initial
   coverage and incident consistency.
3. `analyzeIncident()` identifies the orders exposed on the stopped line.
4. `generateRecoveryPlans()` runs the deterministic Service, Cost and Stability
   policies.
5. `validateSchedule()` rejects any hard-constraint violation.
6. `calculateMetrics()` derives comparable operational and illustrative-cost
   KPI from each complete schedule.
7. `recommendPlan()` reads incident context and the three completed plans and
   suggests one existing route. It cannot alter assignments, feasibility or
   KPI.
8. The app renders the same state as decision cards/Gantt or as an isometric
   workshop twin. All line telemetry, ambient sensors and operators are
   simulated.
9. The manager can inspect another route and explicitly approve it.
10. `approveRecoveryPlan()` returns a new approved object and in-memory audit
    event.
11. Reset creates the original scenario again.

The canonical incident produces the Cost suggestion with 79% displayed model
confidence. Selecting Service instead is a visible demonstration that the
manager, not the model, owns the final decision.

## Deployment boundary

### What the current build can access

```text
Bundled synthetic JavaScript data
              |
              v
Deterministic planner + local embedded model
              |
              v
Local browser or public static browser session
```

### What it cannot access

- PLCs, sensors, drives or machines;
- production networks;
- ERP, MES or APS databases;
- customer systems;
- external AI, telemetry or cloud inference services.

The “Approve” action records a simulated decision in memory. It does not create
a production order or transmit a command. The displayed line and ambient
signals are state-reactive synthetic values, not sensor readings.

## Proposed read-only pilot architecture

```text
ERP/MES/CSV read-only export
            |
            v
Schema + quality validation ------> rejected-record report
            |
            v
Versioned workshop snapshot + versioned constraint model
            |
            v
Deterministic CableTwin planner -> three routes + KPI + explanations
            |                                      |
            |                                      v
            |                         On-site recommender inference
            |                                      |
            +----------------------+---------------+
                                   v
                    Production-manager review and approval
                                   |
                   +---------------+---------------+
                   |                               |
                   v                               v
       versioned pilot audit             comparison only
                                         no automatic write-back

On-site model path, only after enough representative history exists:
real incident history -> local training -> separate holdout -> model registry
```

The first 5–10 historical replays validate the workflow, data and constraints;
they are not automatically a sufficient training set. If retraining becomes
appropriate, it runs inside the site's approved environment so data never
leaves the plant. A failed model gate falls back to the deterministic
three-route comparison.

## What a real deployment would add

| Capability | Reason |
| --- | --- |
| Import adapters and schema mapping | Connect approved read-only sources |
| Persistent versioned store | Preserve inputs, constraints, routes, model suggestion and decisions |
| Authentication and role-based access | Separate configuration, review and approval |
| Constraint administration | Manage effective dates and planner sign-off |
| Model registry and rollback | Version training data, policy, weights, validation and fallback |
| Solver and performance layer | Support larger workshops and time limits |
| Uncertainty / sensitivity analysis | Replan safely when incident duration changes |
| Monitoring and observability | Detect failure, latency, stale data, drift and unusual outputs |
| Security controls | Encryption, retention, access audit and incident response |
| Controlled export | Share an approved plan without direct machine control |

## Non-functional targets for a pilot

- **Feasibility:** zero violations of planner-validated hard constraints.
- **Reproducibility:** same input, rule and model versions produce the same
  result.
- **Traceability:** every route retains data, constraint, objective, model and
  decision lineage.
- **Human control:** learned suggestions remain contestable and require explicit
  approval.
- **Availability:** a documented manual process and deterministic planner
  fallback remain available.
- **Performance:** a response-time target is agreed after measuring realistic
  problem size.
- **Security:** least-privilege, read-only access, minimized data and on-site
  retraining.
- **Usability:** a manager can understand the main trade-off without
  algorithmic vocabulary.
